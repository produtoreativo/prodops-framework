# Ubiquitous Language — Payments Domain

Use these terms in code, DTOs, logs, tests, and PRs.
Never mix synonyms (e.g., do not use "payment" and "invoice" interchangeably).

---

## Glossary

**Invoice**
Invoice created for an order. Aggregate root of this context. Has its own lifecycle.
In code: class `InvoiceRecord`, field `invoiceId`, `InvoiceRepository`,
`InvoiceService`.

**TenantId**
Identifier of the customer/merchant that originated the invoice. All data access is
scoped by `tenantId`. In code: field `tenantId` in all records and queries.

**OrderId**
Reference to the order in the upstream system (Checkout). An invoice belongs to exactly
one order. Payments **stores** `orderId` as an opaque reference — it does not validate
business rules of the order.

**InvoiceId**
Unique identifier of the invoice generated by the Payments API. Prefix `inv_` followed by ULID.
Example: `inv_01HWZXYZ...`. In code: `const invoiceId = \`inv_\${ulid()}\``.

**ExternalReference**
The internal `invoiceId` sent to the provider as a reference. Used to correlate
webhooks back to the invoice. In code: `externalReference: invoiceId` in `InvoiceRecord`.
Never exposes the provider's internal structure.

**Provider**
External payment service. Available providers: `ASAAS`, `ITAU`.
In code: type `PaymentProvider`, field `provider` in the invoice.

**Webhook**
Asynchronous status-change notification sent by the Provider. Processed in
`InvoiceService.processProviderWebhook()`. Can be `PAYMENT_CONFIRMED`,
`PAYMENT_DELETED`, `PAYMENT_AUTHORIZED`, among others.

**PaymentMethod / BillingType**
Payment method: `BOLETO`, `PIX`, `CREDIT_CARD`, `UNDEFINED`.
In code: field `billingType` in `CreateInvoiceDto` and `InvoiceRecord`.

**Status**
Current state of the invoice in its lifecycle:
`CREATED → PROVIDER_PENDING → OPEN → CONFIRMED | RECEIVED | FAILED | CANCELLED`
In code: type `InvoiceStatus` in `invoice.types.ts`.

---

## Lifecycle

```
CREATED → PROVIDER_PENDING → OPEN → CONFIRMED
                                  → RECEIVED
                          → FAILED
          → CANCEL_REQUESTED → CANCELLED
                             → CANCEL_RECONCILIATION_REQUIRED
```

---

## Naming rules

- "invoice" — never "payment", "charge", "bill"
- "confirm" — never "approve", "authorize" or "pay"
- "tenant" — never "merchant", "client" or "user"
- "provider" — never "gateway" or "Asaas" in the domain layer
- "externalReference" — never "asaasId" or "providerCode"
- "orderId" — never "purchaseId" or "transactionId"

Apply to: class/method/variable names, DTO fields, log messages,
test descriptions, PR titles, and commit messages.
